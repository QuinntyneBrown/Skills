# L2 - Detailed Requirements

## L2-001: Create a New Skill
**Traces to:** L1-001

An authenticated user must be able to create a new skill by providing a name, description, prompt/instruction body, tags, and optional configuration parameters. The system must assign a unique identifier and record creation metadata (timestamp, author).

**Acceptance Criteria:**
1. Given an authenticated user, when they submit a valid skill creation form with name, description, body, and at least one tag, then the system must persist the skill and return a unique skill ID.
2. Given an authenticated user, when they submit a skill creation request with a missing required field (name or body), then the system must reject the request with a 400 status and a message identifying the missing field(s).
3. Given an authenticated user, when they create a skill, then the system must record the creation timestamp (UTC) and the author's user ID on the skill record.
4. Given an authenticated user, when they create a skill with a name that exceeds 200 characters, then the system must reject the request with a validation error.
5. Given an authenticated user, when they create a skill with a body that exceeds 500,000 characters, then the system must reject the request with a validation error.

---

## L2-002: Read a Single Skill
**Traces to:** L1-001

An authenticated user must be able to retrieve a skill by its unique identifier. The response must include all skill fields including name, description, body, tags, metadata, and version information.

**Acceptance Criteria:**
1. Given an authenticated user with read access to a skill, when they request the skill by ID, then the system must return the full skill record including name, description, body, tags, created_at, updated_at, author, and current version number.
2. Given an authenticated user, when they request a skill ID that does not exist, then the system must return a 404 status with a descriptive error message.
3. Given an authenticated user without read access to a skill, when they request that skill by ID, then the system must return a 403 status.

---

## L2-003: List Skills
**Traces to:** L1-001

An authenticated user must be able to retrieve a paginated list of skills they are authorized to view.

**Acceptance Criteria:**
1. Given an authenticated user, when they request the skill list without parameters, then the system must return the first page of skills (default page size 25) sorted by updated_at descending.
2. Given an authenticated user, when they request a specific page and page size, then the system must return the corresponding slice of results along with total count, current page, and total pages metadata.
3. Given an authenticated user, when the requested page exceeds the total number of pages, then the system must return an empty results array with correct pagination metadata.
4. Given an authenticated user, when they request the skill list, then only skills the user is authorized to view must be included in results.

---

## L2-004: Update an Existing Skill
**Traces to:** L1-001

An authenticated user with write access must be able to update any mutable field of a skill (name, description, body, tags, configuration). The system must update the modification timestamp and create a new version entry.

**Acceptance Criteria:**
1. Given an authenticated user with write access, when they submit valid updates to a skill's name, description, body, or tags, then the system must persist the changes, update the updated_at timestamp, and increment the version number.
2. Given an authenticated user with write access, when they submit an update with invalid data (e.g., empty name), then the system must reject the request with a 400 status and not modify the skill.
3. Given an authenticated user without write access, when they attempt to update a skill, then the system must return a 403 status.
4. Given an authenticated user, when they attempt to update a skill that does not exist, then the system must return a 404 status.
5. Given two users simultaneously updating the same skill, when the second update is submitted, then the system must detect the conflict (via optimistic concurrency) and return a 409 status to the second user.

---

## L2-005: Delete a Skill
**Traces to:** L1-001

An authenticated user with delete permission must be able to delete a skill. Deletion must be a soft delete (marking the record as deleted) to allow recovery.

**Acceptance Criteria:**
1. Given an authenticated user with delete permission, when they delete a skill, then the system must mark the skill as deleted (soft delete) with a deleted_at timestamp, and the skill must no longer appear in list or search results.
2. Given an authenticated user without delete permission, when they attempt to delete a skill, then the system must return a 403 status.
3. Given an authenticated user with delete permission, when they attempt to delete a skill that does not exist, then the system must return a 404 status.
4. Given a soft-deleted skill, when an admin requests recovery, then the system must restore the skill to active status with all data intact.

---

## L2-006: Database Schema and Integrity
**Traces to:** L1-002

The database must enforce referential integrity, uniqueness constraints, and indexing to support efficient queries on skills, users, and related entities.

**Acceptance Criteria:**
1. Given the database schema, when a skill is inserted, then a unique constraint on (owner_id, name) must prevent duplicate skill names per user.
2. Given the database schema, when a query filters skills by tag, then an index on the tags column must ensure query response time under 100ms for datasets up to 1 million skills.
3. Given the database schema, when a skill is referenced by a version record, then a foreign key constraint must prevent orphaned version records.
4. Given a database failure during a write operation, when the transaction is interrupted, then no partial data must be committed (ACID compliance).

---

## L2-007: Database Migrations
**Traces to:** L1-002

All database schema changes must be managed through versioned, reversible migrations.

**Acceptance Criteria:**
1. Given a new deployment, when migrations are executed, then the schema must be created from scratch in the correct order without errors.
2. Given an existing deployment, when a new migration is applied, then only the incremental change must be executed, leaving existing data intact.
3. Given an applied migration, when a rollback is requested, then the migration must be reversible without data loss for non-destructive changes.

---

## L2-008: User Registration
**Traces to:** L1-003

New users must be able to register for an account using an email address and password, or via an OAuth provider.

**Acceptance Criteria:**
1. Given a visitor, when they submit a registration form with a valid email and a password meeting complexity requirements (minimum 12 characters, at least one uppercase, one lowercase, one digit, one special character), then the system must create the account and send a verification email.
2. Given a visitor, when they submit a registration with an email already in use, then the system must return a generic error that does not reveal whether the email exists (to prevent enumeration).
3. Given a visitor, when they register via a supported OAuth provider (e.g., GitHub, Google), then the system must create or link the account and establish a session.
4. Given a newly registered user, when they have not verified their email, then the system must restrict access to read-only operations until verification is complete.

---

## L2-009: User Login and Session Management
**Traces to:** L1-003

Authenticated sessions must be established via secure token-based authentication with appropriate expiration and refresh mechanisms.

**Acceptance Criteria:**
1. Given a registered user, when they submit valid credentials, then the system must issue a short-lived access token (maximum 15 minutes) and a refresh token (maximum 7 days).
2. Given a user with an expired access token and a valid refresh token, when they request a token refresh, then the system must issue a new access token and rotate the refresh token.
3. Given a user, when they submit invalid credentials 5 times within 10 minutes, then the system must lock the account for 15 minutes and notify the user via email.
4. Given a user, when they log out, then the system must invalidate the current access and refresh tokens immediately.
5. Given a user session, when the user changes their password, then all other active sessions for that user must be invalidated.

---

## L2-010: Role-Based Access Control
**Traces to:** L1-003

The system must enforce role-based access control (RBAC) with at least the following roles: admin, member, and viewer.

**Acceptance Criteria:**
1. Given a user with the admin role, when they perform any operation (CRUD on any skill, user management), then the system must permit the action.
2. Given a user with the member role, when they create, read, update, or delete their own skills, then the system must permit the action.
3. Given a user with the member role, when they attempt to modify another user's skill, then the system must return a 403 status.
4. Given a user with the viewer role, when they attempt to create, update, or delete any skill, then the system must return a 403 status.
5. Given a user with the viewer role, when they read skills marked as public or shared with them, then the system must permit the action.

---

## L2-011: Skill Sharing and Visibility
**Traces to:** L1-003

Users must be able to control the visibility of their skills (private, shared, public) and share skills with specific users.

**Acceptance Criteria:**
1. Given a skill owner, when they set a skill's visibility to private, then only the owner and admins must be able to access it.
2. Given a skill owner, when they share a skill with specific users by email or user ID, then those users must gain read access to the skill.
3. Given a skill owner, when they set a skill's visibility to public, then any authenticated user must be able to read the skill.
4. Given a user with shared access, when they attempt to modify the skill, then the system must return a 403 status unless explicitly granted write access.

---

## L2-012: Dark Mode UI Theme
**Traces to:** L1-004

The web UI must render in a dark color scheme by default, with sufficient contrast ratios for accessibility compliance.

**Acceptance Criteria:**
1. Given a user loading the web UI, when the page renders, then the background must be a dark color (#1a1a2e or similar) and text must be a light color (#e0e0e0 or similar) with a minimum contrast ratio of 4.5:1 (WCAG AA).
2. Given the dark-mode UI, when interactive elements (buttons, links, inputs) are displayed, then they must use accent colors that meet a minimum contrast ratio of 3:1 against adjacent colors.
3. Given the dark-mode UI, when syntax-highlighted code is displayed in the editor, then all token colors must meet a minimum contrast ratio of 4.5:1 against the editor background.

---

## L2-013: Code/Text Editor for Skill Authoring
**Traces to:** L1-004

The web UI must include a high-quality embedded code editor for authoring and editing skill prompt/instruction content.

**Acceptance Criteria:**
1. Given an authenticated user editing a skill, when they open the editor, then the editor must render with syntax highlighting for Markdown content.
2. Given an authenticated user in the editor, when they type, then the editor must support auto-indentation and bracket matching.
3. Given an authenticated user in the editor, when they press standard keyboard shortcuts (Ctrl+Z/Cmd+Z for undo, Ctrl+Shift+Z/Cmd+Shift+Z for redo, Ctrl+S/Cmd+S for save), then the editor must respond with the expected action.
4. Given an authenticated user in the editor, when the skill body exceeds 10,000 lines, then the editor must remain responsive (no perceptible input lag exceeding 50ms).
5. Given an authenticated user in the editor, when they search within the document (Ctrl+F/Cmd+F), then the editor must highlight all matches and support find-and-replace.
6. Given an authenticated user in the editor, when they edit content, then the editor must display a line count, cursor position (line:column), and character count in a status bar.

---

## L2-014: Skill Management Dashboard
**Traces to:** L1-004

The web UI must provide a dashboard view listing all skills accessible to the user, with quick actions for common operations.

**Acceptance Criteria:**
1. Given an authenticated user, when they navigate to the dashboard, then they must see a list of their skills showing name, description (truncated), tags, last updated date, and visibility status.
2. Given an authenticated user on the dashboard, when they click a skill, then the system must navigate to the skill detail/edit view.
3. Given an authenticated user on the dashboard, when they click "New Skill," then the system must navigate to the skill creation form with the editor.
4. Given an authenticated user on the dashboard, when they select one or more skills and choose "Delete," then the system must prompt for confirmation before executing deletion.
5. Given an authenticated user on the dashboard, when skill data is loading, then the system must display skeleton loaders (not a blank screen or spinner alone).

---

## L2-015: CLI Authentication
**Traces to:** L1-005

The CLI must support authentication via API key or interactive login flow that issues a token stored securely on the local filesystem.

**Acceptance Criteria:**
1. Given a user running the CLI for the first time, when they execute the login command, then the system must open a browser-based OAuth flow and store the resulting token in a platform-appropriate secure location (e.g., OS keychain or encrypted file with 600 permissions).
2. Given a user with a valid stored token, when they execute any CLI command, then the system must use the stored token without prompting for re-authentication.
3. Given a user with an expired token, when they execute a CLI command, then the system must attempt token refresh automatically and only prompt for re-login if the refresh fails.
4. Given a user, when they execute the logout command, then the system must delete the stored credentials from the local filesystem.
5. Given a user, when they set the environment variable `CLAUDE_SKILLS_API_KEY`, then the CLI must use that key for authentication instead of the stored token.

---

## L2-016: CLI Skill Operations
**Traces to:** L1-005

The CLI must support creating, reading, listing, updating, and deleting skills via subcommands.

**Acceptance Criteria:**
1. Given an authenticated CLI user, when they run `skills create --name "My Skill" --body-file ./skill.md --tags "tag1,tag2"`, then the system must create the skill and print the new skill ID to stdout.
2. Given an authenticated CLI user, when they run `skills get <skill-id>`, then the system must print the full skill record to stdout in a human-readable format (default) or JSON (with `--json` flag).
3. Given an authenticated CLI user, when they run `skills list`, then the system must print a paginated table of skills with columns for ID, name, updated_at, and visibility.
4. Given an authenticated CLI user, when they run `skills update <skill-id> --name "New Name"`, then the system must update the specified fields and print a confirmation.
5. Given an authenticated CLI user, when they run `skills delete <skill-id>`, then the system must prompt for confirmation (unless `--yes` flag is passed) and print the result.
6. Given an authenticated CLI user, when they run `skills edit <skill-id>`, then the system must open the skill body in the user's `$EDITOR` and submit the changes on save-and-close.
7. Given any CLI command that fails, when the error occurs, then the system must print a descriptive error message to stderr and exit with a non-zero exit code.

---

## L2-017: CLI Output Formats
**Traces to:** L1-005

The CLI must support multiple output formats to enable both human use and scripting/automation.

**Acceptance Criteria:**
1. Given an authenticated CLI user, when they append `--json` to any read or list command, then the output must be valid JSON written to stdout.
2. Given an authenticated CLI user, when they append `--quiet` to any mutating command, then only the affected resource ID must be written to stdout (no confirmation text).
3. Given an authenticated CLI user piping CLI output, when stdout is not a TTY, then the system must suppress color codes and progress indicators automatically.

---

## L2-018: RESTful API Design
**Traces to:** L1-006

The API must follow RESTful conventions with versioned endpoints, standard HTTP methods, and consistent response envelopes.

**Acceptance Criteria:**
1. Given the API, when a client makes a request, then all endpoints must be prefixed with `/api/v1/`.
2. Given the API, when a client creates a resource, then the endpoint must use POST and return 201 with the created resource in the response body.
3. Given the API, when a client reads a resource, then the endpoint must use GET and return 200 with the resource in the response body.
4. Given the API, when a client updates a resource, then the endpoint must use PATCH and return 200 with the updated resource.
5. Given the API, when a client deletes a resource, then the endpoint must use DELETE and return 204 with no body.
6. Given the API, when any error occurs, then the response body must follow the format: `{ "error": { "code": "<ERROR_CODE>", "message": "<human-readable message>", "details": [...] } }`.

---

## L2-019: API Documentation
**Traces to:** L1-006

The API must be documented with an OpenAPI (Swagger) specification that is auto-generated from the codebase and accessible via a public endpoint.

**Acceptance Criteria:**
1. Given the running application, when a client requests `/api/docs`, then the system must serve an interactive API documentation UI (e.g., Swagger UI or Redoc).
2. Given the running application, when a client requests `/api/v1/openapi.json`, then the system must return the current OpenAPI 3.x specification.
3. Given any API endpoint change, when the application is built, then the OpenAPI specification must reflect the change without manual editing.

---

## L2-020: API Rate Limiting
**Traces to:** L1-006

The API must enforce rate limits to protect against abuse and ensure fair usage.

**Acceptance Criteria:**
1. Given an authenticated user, when they exceed 100 requests per minute to any endpoint, then the system must return a 429 status with a `Retry-After` header indicating when the user may retry.
2. Given an unauthenticated request, when it exceeds 20 requests per minute from a single IP, then the system must return a 429 status.
3. Given any API response, when rate limiting is active, then the response must include `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset` headers.

---

## L2-021: Structured Application Logging
**Traces to:** L1-007

All application logs must be structured (JSON format), include correlation IDs for request tracing, and follow consistent severity levels.

**Acceptance Criteria:**
1. Given any API request, when it is processed, then the system must log a structured JSON entry containing: timestamp (ISO 8601), correlation_id, method, path, status_code, duration_ms, and user_id (if authenticated).
2. Given any unhandled exception, when it occurs, then the system must log the exception at ERROR level with stack trace, correlation_id, and request context.
3. Given the application, when logs are emitted, then they must use severity levels: DEBUG, INFO, WARN, ERROR, FATAL, and the production default must be INFO.
4. Given a log entry, when it involves user data, then the system must never log passwords, tokens, or personally identifiable information (PII) in plaintext.

---

## L2-022: Audit Logging
**Traces to:** L1-007

All mutating operations on skills and user accounts must generate immutable audit log entries.

**Acceptance Criteria:**
1. Given any create, update, or delete operation on a skill, when the operation completes, then the system must write an audit log entry containing: action, resource_type, resource_id, user_id, timestamp, and a summary of changes (before/after for updates).
2. Given any authentication event (login, logout, failed login, password change), when the event occurs, then the system must write an audit log entry.
3. Given the audit log, when queried, then entries must be immutable (append-only) and must not be deletable through the application.
4. Given an admin, when they query the audit log via the API, then the system must support filtering by user_id, resource_type, action, and date range.

---

## L2-023: Health Checks and Metrics
**Traces to:** L1-007

The system must expose health check and metrics endpoints for infrastructure monitoring.

**Acceptance Criteria:**
1. Given the running application, when a monitoring system requests `/health`, then the system must return 200 with a JSON body indicating status of each dependency (database, cache, external services).
2. Given the running application, when a monitoring system requests `/metrics`, then the system must return Prometheus-compatible metrics including request_count, request_duration_seconds, active_connections, and error_rate.
3. Given a database connectivity failure, when `/health` is requested, then the system must return 503 with a body identifying the degraded dependency.

---

## L2-024: Input Validation and Sanitization
**Traces to:** L1-008

All user input must be validated on the server side before processing. Output must be encoded to prevent injection attacks.

**Acceptance Criteria:**
1. Given any API request with a request body, when the body contains fields not defined in the schema, then the system must strip unknown fields and process only known fields.
2. Given any API request, when the body contains HTML or script tags in text fields, then the system must sanitize or escape them before storage and always encode on output.
3. Given any API request, when a path or query parameter contains SQL injection patterns, then the system must reject the request or neutralize the input via parameterized queries.
4. Given any file upload (e.g., skill body from file), when the file exceeds 5MB, then the system must reject the upload with a 413 status.

---

## L2-025: CSRF, CORS, and Secure Headers
**Traces to:** L1-008

The system must implement CSRF protection, a strict CORS policy, and security-related HTTP headers.

**Acceptance Criteria:**
1. Given any state-mutating request from the web UI, when the request is submitted, then it must include a valid CSRF token, and the server must reject requests with missing or invalid tokens with a 403 status.
2. Given the API, when a cross-origin request is received, then the server must only allow origins explicitly configured in the CORS allowlist.
3. Given any HTTP response, when headers are set, then the response must include: `Strict-Transport-Security` (max-age >= 31536000), `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Content-Security-Policy` with a restrictive policy, and `Referrer-Policy: strict-origin-when-cross-origin`.

---

## L2-026: Data Encryption
**Traces to:** L1-008

All data must be encrypted in transit and sensitive data must be encrypted at rest.

**Acceptance Criteria:**
1. Given any client-server communication, when data is transmitted, then TLS 1.2 or higher must be used with no fallback to unencrypted connections.
2. Given user passwords, when stored in the database, then they must be hashed using bcrypt (or Argon2id) with a work factor sufficient to require >= 250ms per hash.
3. Given API keys and refresh tokens, when stored in the database, then they must be stored as hashed values, not plaintext.

---

## L2-027: API Response Time Targets
**Traces to:** L1-009

The API must meet defined response-time targets under normal operating conditions.

**Acceptance Criteria:**
1. Given normal load (up to 100 concurrent users), when a single skill is requested (GET by ID), then the 95th percentile response time must be under 200ms.
2. Given normal load, when a skill list is requested with default pagination, then the 95th percentile response time must be under 500ms.
3. Given normal load, when a skill is created or updated, then the 95th percentile response time must be under 500ms.
4. Given peak load (up to 1,000 concurrent users), when any API endpoint is called, then the 99th percentile response time must be under 2,000ms.

---

## L2-028: Horizontal Scalability
**Traces to:** L1-009

The system architecture must support horizontal scaling of stateless application instances behind a load balancer.

**Acceptance Criteria:**
1. Given the application, when deployed, then all application instances must be stateless (no local session storage, no local file dependencies for runtime data).
2. Given two or more application instances behind a load balancer, when a user's requests are routed to different instances, then the user's experience must be identical (session state managed externally).
3. Given increased load, when a new application instance is added, then it must begin serving requests within 30 seconds of startup with no manual configuration.

---

## L2-029: Database Connection Pooling and Caching
**Traces to:** L1-009

The system must use connection pooling for database access and caching for frequently accessed data.

**Acceptance Criteria:**
1. Given the application, when it connects to the database, then it must use a connection pool with configurable min/max connections and idle timeout.
2. Given a skill that has been read recently, when the same skill is requested again within the cache TTL, then the system must serve it from cache without a database query.
3. Given a skill that is updated, when the update is committed, then the corresponding cache entry must be invalidated immediately.

---

## L2-030: Mobile Layout (Extra-Small Viewports)
**Traces to:** L1-010

The web UI must provide a fully functional single-column layout on viewports below 576px.

**Acceptance Criteria:**
1. Given a viewport width < 576px, when the user views the dashboard, then the layout must switch to a single-column card list with full-width skill cards.
2. Given a viewport width < 576px, when the user opens the skill editor, then the editor must occupy the full viewport width with a collapsible toolbar.
3. Given a viewport width < 576px, when the user navigates, then the navigation menu must be accessible via a hamburger menu icon.

---

## L2-031: Tablet and Desktop Layouts
**Traces to:** L1-010

The web UI must provide optimized layouts for tablet (>=768px) and desktop (>=992px) viewports.

**Acceptance Criteria:**
1. Given a viewport width >= 768px and < 992px, when the user views the dashboard, then the layout must display skills in a two-column grid.
2. Given a viewport width >= 992px, when the user views the dashboard, then the layout must display a sidebar navigation alongside the main content area.
3. Given a viewport width >= 1200px, when the user edits a skill, then the editor must support a split-pane view with the editor on the left and a live preview on the right.

---

## L2-032: API Error Responses
**Traces to:** L1-011

All API errors must return consistent, informative error responses with appropriate HTTP status codes.

**Acceptance Criteria:**
1. Given a validation error, when the API responds, then it must return 400 with an error body listing each invalid field and the reason for rejection.
2. Given an authentication failure, when the API responds, then it must return 401 with a generic "Unauthorized" message (no detail about which credential was wrong).
3. Given an unexpected server error, when the API responds, then it must return 500 with a generic message and a correlation_id for debugging, without exposing stack traces or internal details.
4. Given a request to a non-existent endpoint, when the API responds, then it must return 404 with a message suggesting valid endpoints or linking to API documentation.

---

## L2-033: Web UI Error Handling
**Traces to:** L1-011

The web UI must display user-friendly error states and support recovery without page reloads where possible.

**Acceptance Criteria:**
1. Given a network failure during a save operation, when the error occurs, then the UI must display an inline error notification with a "Retry" action and must not discard the user's unsaved changes.
2. Given a 401 or 403 response, when the error occurs during a UI operation, then the UI must redirect the user to the login page with a message indicating the session has expired.
3. Given a 500 response, when the error occurs, then the UI must display a friendly error message with the correlation_id and a suggestion to contact support.
4. Given the editor, when the browser loses network connectivity, then the editor must continue to function offline and queue changes for submission when connectivity is restored.

---

## L2-034: CLI Error Handling
**Traces to:** L1-011

The CLI must provide clear error messages on failure and use standard exit codes.

**Acceptance Criteria:**
1. Given any CLI command that fails due to a network error, when the error occurs, then the CLI must print a message suggesting the user check connectivity and retry, and exit with code 1.
2. Given any CLI command that fails due to an authentication error, when the error occurs, then the CLI must print a message suggesting re-authentication (`skills login`) and exit with code 2.
3. Given any CLI command with invalid arguments, when the error occurs, then the CLI must print the specific validation error and the correct usage syntax, and exit with code 64.

---

## L2-035: Full-Text Search
**Traces to:** L1-012

Users must be able to perform full-text search across skill names, descriptions, and body content.

**Acceptance Criteria:**
1. Given an authenticated user, when they submit a search query via the API (`GET /api/v1/skills?q=<query>`), then the system must return skills where the query matches the name, description, or body, ranked by relevance.
2. Given a search query, when results are returned, then the response must include highlighted snippets showing where the match occurred.
3. Given a dataset of 100,000 skills, when a full-text search is executed, then the 95th percentile response time must be under 500ms.

---

## L2-036: Filtering and Sorting
**Traces to:** L1-012

Users must be able to filter skill lists by tag, author, visibility, and date range, and sort by any displayed column.

**Acceptance Criteria:**
1. Given an authenticated user, when they filter skills by one or more tags (`?tags=tag1,tag2`), then the system must return only skills matching all specified tags (AND logic).
2. Given an authenticated user, when they filter by date range (`?created_after=<date>&created_before=<date>`), then the system must return only skills created within the range.
3. Given an authenticated user, when they sort by any supported field (`?sort=name&order=asc`), then the results must be returned in the specified order.
4. Given the web UI, when a user applies filters, then the URL query string must update to reflect the current filter state, enabling bookmarking and sharing of filtered views.

---

## L2-037: Skill Version History
**Traces to:** L1-013

Every change to a skill's content must create a new version record preserving the previous state.

**Acceptance Criteria:**
1. Given a skill that is updated, when the update is persisted, then a version record must be created containing the previous content, the user who made the change, and the timestamp.
2. Given an authenticated user, when they request the version history of a skill (`GET /api/v1/skills/<id>/versions`), then the system must return a paginated list of versions in reverse chronological order.
3. Given an authenticated user, when they request a specific version (`GET /api/v1/skills/<id>/versions/<version>`), then the system must return the full skill content as it existed at that version.

---

## L2-038: Version Comparison and Restoration
**Traces to:** L1-013

Users must be able to compare two versions of a skill side-by-side and restore a previous version.

**Acceptance Criteria:**
1. Given an authenticated user, when they request a diff between two versions (`GET /api/v1/skills/<id>/versions/<v1>/diff/<v2>`), then the system must return a structured diff showing added, removed, and changed lines.
2. Given the web UI, when a user views a version diff, then the UI must render a side-by-side or unified diff view with color-coded additions (green) and deletions (red) that meets the dark-mode contrast requirements.
3. Given an authenticated user with write access, when they restore a previous version, then the system must create a new version (not overwrite history) with the restored content and record the restoration action in the audit log.
